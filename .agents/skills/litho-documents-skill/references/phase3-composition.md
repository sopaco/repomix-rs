# 阶段三：文档编排生成详细指南

## 核心原则

1. **基于研究，不臆测**：所有描述必须有阶段二研究报告中的代码证据支撑
2. **Mermaid 图表必须可渲染**：遵循语法规范（见 `doc-templates.md`）
3. **结构完整**：每份文档必须包含规定的章节，不能省略核心章节
4. **语言一致**：使用用户指定的输出语言（默认中文）
5. **代码引用密度**：每个模块文档必须引用至少 3 个具体文件路径，关键组件表格每行必须有文件路径列
6. **分章节增量写入**：大型文档分多次写入，避免单次响应过长
7. **中间产物持久化**：研究报告写入 `.litho-agent/` 临时目录，编排阶段按需读取
8. **跨模块协作叙述**：每个模块文档包含模块在核心业务流程中的角色描述
9. **叙述性写作风格**：文档必须面向人类阅读友好，而非冷冰冰的 PPT 式结构化文字（见下方详细指南）

---

## ⚠️ 叙述性写作风格指南（P0 关键优化）

### 问题
Agent 生成的文档容易呈现"PPT式"风格：章节标题后直接跳到表格/列表，缺乏上下文解读和叙述性描述。这种文档对人类读者来说像在看幻灯片提纲，而不是阅读一篇有深度的技术文章。

### 核心要求

#### 1. 每个章节开头必须有叙述性 summary
不要在章节标题后直接甩出表格或列表。先用 2-4 句话解释"这个章节在说什么、为什么重要、读者应该关注什么"。

**❌ 错误示例**：
```
## 它能做什么
- 智能代码分析
- C4 模型文档生成
- 外部知识增强
- AI 自主推理
```

**✅ 正确示例**：
```
## 它能做什么

Litho 的核心能力可以概括为"从源码到文档的全自动流水线"。简单来说，你给它一个项目的源码目录，它就会自动帮你分析代码结构、识别核心模块、梳理依赖关系，然后利用大语言模型把所有这些信息组织成一套专业的 C4 模型架构文档。

**智能代码分析**——Litho 不是简单地扫描文件名，而是真正去解析代码结构。它内置了14种语言的处理器……
```

#### 2. 表格前后必须有解读段落
不要孤立地放置表格。在表格之前解释"为什么这个表格重要、你应该从中读出什么"，在表格之后总结"从这些数据中我们能得出什么结论"。

**❌ 错误示例**：
```
## 技术栈
| 技术 | 选择 |
|------|------|
| Rust | 语言 |
| Tokio | 异步 |
```

**✅ 正确示例**：
```
## 技术选型背后的思考

Litho 选择 Rust 作为开发语言，这不仅仅是追求"高性能"这么简单。作为一个需要大量调用外部 LLM API 的 IO 密集型工具，Rust 配合 Tokio 异步运行时可以高效地管理并发请求和缓存读写……

| 技术领域 | 具体选择 | 为什么这样选 |
|---------|---------|------------|
| 语言与运行时 | Rust (edition 2024) | 内存安全 + 高性能，配合 Tokio 处理密集 IO |
| … | … | … |
```

#### 3. 设计决策必须讲"为什么"——选择、放弃、原因
架构决策不只是"选了什么"。要讲清楚"选了什么、放弃了什么替代方案、为什么做出这个选择"。

**❌ 错误示例**：
```
| 决策 | 依据 |
|------|------|
| Memory scope 隔离 | MemoryScope/ScopedKeys常量 |
```

**✅ 正确示例**：
```
| 决策 | 选择了什么 | 放弃了什么 | 为什么 |
|------|---------|---------|------|
| 数据传递机制 | Memory scope 隔离 | 全局 HashMap 共享 | 防止阶段间数据互相覆盖，让数据流可追踪 |
```

#### 4. 用类比和比喻建立理解桥梁
技术概念用日常类比来解释，帮助读者快速建立理解框架。

**示例**：
- Memory scope 隔离 → "快递站的分区域管理，每个区域只收自己的包裹"
- Agent 统一抽象 → "标准化的工作台，工人只需要声明'我要什么工具'，工作台自动准备"
- Pipeline → "汽车生产线，原材料进厂 → 各车间接力加工 → 质检出厂"
- CacheManager → "省钱利器，同样的推理不做第二次"
- StepForwardAgent → "所有 Agent 的灵魂/岗位说明书"

#### 5. 模块深度文档以"这个模块在做什么"开头
不要以冷冰冰的模块名和路径列表开头。先用一句通俗的话解释模块的存在意义。

**❌ 错误示例**：
```
# Research 模块深度探索

## 核心功能点
1. C1宏观分析
2. C2中观分析
3. C3微观分析
```

**✅ 正确示例**：
```
# Research 模块深度探索

## 这个模块在做什么

Research 模块是 Litho 的"大脑"——它负责对项目进行真正的深度理解。预处理阶段只是把源码拆解成了结构化的信息碎片，而 Research 模块的工作是把这些碎片拼成一个完整的图景……

## 七个 Agent 的分工

Research 模块内部有7个专业Agent，它们各自负责不同的分析维度，但按照固定的层级顺序执行：
| Agent | 所属层级 | 它负责回答什么问题 |
|-------|---------|-----------------|
| SystemContextResearcher | C1宏观 | 这个系统解决什么问题、服务于谁？ |
```

#### 6. 工作流用比喻开头，再展开细节
工作流文档不要直接列出步骤1→2→3→4。先用比喻建立整体认知，然后才展开每一步的细节。

**示例**：
```
Litho 的工作流可以用一个简单的比喻来理解：它就像一条汽车生产线——原材料（源码）进厂后，先在预处理车间拆解分类……
```

#### 7. 参数表格增加"含义解读"列
边界接口文档的参数不要只有"参数名/类型/默认值/说明"，应该增加一列解释这个参数的**实际含义和设计意图**，而不是只描述它是什么。

**❌ 错误示例**：
```
| 参数 | 类型 | 默认值 | 说明 |
|------|------|-------|------|
| llm-provider | String | openai | LLM Provider |
```

**✅ 正确示例**：
```
| 参数 | 默认值 | 含义解读 |
|------|-------|---------|
| llm-provider | openai | 用哪家 LLM 服务——不同 Provider 的 API 格式和定价差异很大，这个参数决定了你的推理能力和成本 |
| model-powerful | 与efficient相同 | 复杂推理和降级备用的模型——建议配一个比 efficient 更强的模型，关键时刻有强力模型兜底 |
```

### 写作风格自检清单（每次写入文档前必查）

- [ ] 每个章节开头是否有 2-4 句话的叙述性 summary？
- [ ] 表格前后是否有解读段落（而非孤立的表格）？
- [ ] 架构决策是否讲了"选择了什么、放弃了什么、为什么"？
- [ ] 是否使用了类比/比喻来解释关键概念？
- [ ] 模块深度文档是否以"这个模块在做什么"开头？
- [ ] 工作流是否先用比喻/叙述建立整体认知？
- [ ] 参数表格是否有"含义解读"而非只有技术描述？
- [ ] 文档读起来像"技术文章"而非"幻灯片提纲"？

## ⚠️ 中间产物持久化策略

### 问题
Agent 单次对话上下文窗口有限。随着分析深入，早期的研究结果可能因上下文压力被遗忘。

### 解决方案
每完成一个研究步骤，将关键发现写入 `.litho-agent/` 临时目录：

```
.litho-agent/
├── preprocessing.md        ← 阶段一产出
├── c1-system-context.md    ← C1 系统上下文
├── c2-domain-modules.md    ← C2 领域模块
├── architecture.md         ← 架构研究
├── workflow.md             ← 工作流研究
├── boundary.md             ← 边界接口
├── database.md             ← 数据库（条件）
└── modules/                ← 模块深度（逐个写入）
    ├── llm.md
    ├── cache.md
    └── ...
```

**编排阶段读取方法**：
- 需要某报告时 → `read_file` 从 `.litho-agent/` 读取
- 不需要时 → 不读取，节省上下文
- 最终输出完成后 → 删除 `.litho-agent/`（可选保留供复查）

**核心优势**：与 deepwiki-rs 的 Memory 作用域机制等效，但使用文件系统而非内存

## ⚠️ 分章节增量写入策略（P0 关键优化）

deepwiki-rs 通过多次独立 LLM 调用生成每个文档。Agent 单次响应容量有限，必须分批写入：

### 大型文档分章节策略

| 文档 | 写入策略 | 分批方式 |
|------|---------|---------|
| `1.概述.md` | 单次写入 | 中等文档，通常可一次完成 |
| `2.架构.md` | **分章节写入** | 第1次：框架+设计理念+架构模式表+技术栈 → 第2次：C4图+模块职责表 → 第3次：关键流程时序图+技术实现+ADR |
| `3.工作流.md` | **分章节写入** | 第1次：框架+核心路径图 → 第2次：主要工作流流程图+时序图 → 第3次：并发模型+错误处理 |
| `4.Deep-Exploration/*.md` | **逐模块写入** | 每个模块一个独立 write_to_file 调用 |
| `5.边界接口.md` | 单次写入 | 通常可一次完成 |
| `6.数据库概览.md` | 单次写入 | 条件触发时一次性完成 |

### 增量写入操作方法
- 第1次：`write_to_file` 写入完整框架（所有章节标题 + Mermaid 图 + 核心表格）
- 第2次+：`replace_in_file` 在各章节的 `{待补充}` 占位处插入详细内容

## Mermaid 图表语法规范（重要！）

### 节点 ID 规则
```
✅ 正确：A, B1, NodeA, myNode
❌ 错误：My Node, node-1, @user, node.1
```

### 节点标签规则（含特殊字符时必须加双引号）
```
✅ 正确：A["用户 (User)"] --> B["系统 @backend"]
❌ 错误：A[用户 (User)] --> B[系统 @backend]
```

### 标签内换行
```
✅ 正确：A["第一行<br/>第二行"]
❌ 错误：A["第一行
第二行"]
```

### Mermaid 图类型选择指南
| 图类型 | 适用场景 | 语法关键词 |
|-------|---------|---------|
| `C4Context` | 系统上下文图（用户+外部系统） | `Person`, `System`, `Rel` |
| `C4Container` | 容器架构图（主要子系统） | `Container`, `Container_Boundary` |
| `C4Component` | 组件图（模块内部） | `Component`, `Rel` |
| `flowchart TD` | 流程图（自上而下） | `-->`, `--\|label\|-->` |
| `flowchart LR` | 流程图（从左到右） | `-->` |
| `sequenceDiagram` | 时序图（跨系统交互） | `participant`, `->>`, `-->>` |
| `graph LR` | 依赖关系图 | `-->` |
| `erDiagram` | 实体关系图（数据库） | `}|--\|{`, `TABLE {` |
