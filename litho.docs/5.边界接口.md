# 5. 边界接口

`repomix-rs` 有三种可见的接入口：终端里的单条命令、被 AI agent 直接驱动的 MCP 工具，以及被别的 Rust crate 引用的库 API。这三条通路共享同一颗 `repomix-core` 的心脏，但各自负责不同层面的交互和协议翻译。

## 5.1 CLI 接口

入口点是 `crates/cli/src/main.rs` 中的 `fn main()`，通过 `clap` derive 生成所有参数解析。

### 用户身份

| 角色 | 关心什么 | 典型操作 |
|---|---|---|
| 本地开发者 | 打包当前项目发给 LLM | `repomix . --style markdown` |
| CI 流水线 | 可重复、无人值守的打包 | `repomix --no-timestamp --style json .` |
| 新手用户 | 快速生成推荐配置 | `repomix --init` |

### 完整选项清单

```text
repomix [OPTIONS] [ROOT]

位置参数:
  ROOT              要打包的目录（缺省为当前目录）

布尔开关:
  --remote <URL>    克隆远程仓库后打包
  --compress        启用 tree-sitter 代码签名压缩
  --remove-comments 删除注释
  --remove-empty-lines 删除空行
  --line-numbers    为文件内容添加行号
  --truncate-base64 截断超长 base64 字符串
  --copy            将输出复制到系统剪贴板
  --init            交互式生成配置文件
  --mcp             以 MCP 服务器模式运行
  --include-diffs   在输出中加入 git diff
  --include-logs    在输出中加入 git log
  -v, -vv           增加日志详细程度

取值参数:
  --style <STYLE>   输出风格: xml|markdown|plain|json（默认 xml）
  --output <PATH>   输出文件路径（默认 repomix-output.<ext>）
  --include <CSV>   逗号分隔的包含 glob 列表
  --ignore <CSV>    逗号分隔的忽略 glob 列表
  --header-text <TEXT>   自定义输出头部文本
  --instruction-file <FILE>   从文件读取指令文本
  --top-files-length <N>     报告中最大的 N 个 token 文件（默认 10）
  --split-output <TOKENS>    按 token 数拆分输出为多个文件（XML 在文件边界切分）
```

CLI 执行策略是一目了然的：`--init` 和 `--mcp` 是完整的平行模式，其余操作统一走 `run::run_pack` 同一条流水线，差异全由 `RepomixConfig` 承载。

### 关键边界行为

- 选项通过 `PartialConfig` 结构传入 `RepomixConfig::load`，与项目级和全局配置文件做层叠加；用户永远不需要知道配置来自哪一层。
- 未提供 `ROOT` 时自动取 `std::env::current_dir()`，这是唯一一个 CLI 版本内部的兜底默认。
- `--remote` 用 `PID + 纳秒时间戳 + 哈希` 组合生成唯一临时目录，并通过 RAII `TempDirGuard` 在进程退出时自动清理，避免 `/tmp` 残骸。

## 5.2 库接口

对外暴露给 Rust crate 使用者的 API 集中在 `repomix-core/lib.rs` 的 re-export 和 `crates/core/src/packager.rs`。

### 核心签名

```rust
// repomix-core
pub async fn pack(
    root_dirs: Vec<PathBuf>,
    config: RepomixConfig,
    progress: Box<dyn ProgressCallback>,
) -> Result<PackResult>

pub struct PackOptions {
    pub root_dirs: Vec<PathBuf>,
    pub config: RepomixConfig,
}
// builder 方法: new, with_config, with_style, with_compress, with_remove_comments,
//              with_line_numbers, with_include_patterns, with_ignore_patterns

pub trait ProgressCallback: Send + Sync {
    fn on_progress(&self, message: &str);
    fn on_complete(&self, message: &str);
    fn on_error(&self, message: &str);
}

pub struct NoopProgress;  // 默认空实现
pub struct PackResult;
```

`PackResult` 的结构直接反映了 `pack()` 的产物：

| 字段 | 含义 |
|---|---|
| `total_files` | 最终输出中的文件数 |
| `total_characters` | 输出总字符数 |
| `total_tokens` | 输出总 token 数（tiktoken-rs 或兜底估算） |
| `file_char_counts` | 每个文件的字符数（Map） |
| `file_token_counts` | 每个文件的 token 数（Map） |
| `top_files_by_tokens` | 按 token 数降序排列的前 N 个文件（N = `top_files_length`） |
| `git_diff_content` | 可选的 `git diff` 文本 |
| `git_diff_token_count` | 对应 token 数 |
| `git_log_content` | 可选的 `git log` 文本 |
| `git_log_token_count` | 对应 token 数 |
| `output_files` | 实际写入磁盘的文件路径列表（拆分模式下有多条） |
| `suspicious_files` | 被 Secretlint 标记并过滤掉的文件 |
| `processed_files` | 最终参与合成的文件列表 |
| `safe_file_paths` | 通过安全检查的文件路径 |
| `skipped_files` | 因 max-size 或其他原因跳过的文件 |

便捷函数提供三条一键调用路径：

```rust
pub async fn pack_directory(dir: &str) -> Result<PackResult>
pub async fn pack_with_config(dir: &str, config: RepomixConfig) -> Result<PackResult>
pub async fn pack_with_options(options: PackOptions) -> Result<PackResult>
```

### 依赖导出规则

`repomix-core` 内部使用 `pub use` 把 `config` 全模块、核心类型和 `OutputStyle` 等符号重新导出，让库用户只需加一条 `repomix-core` 依赖即可访问常用类型。

## 5.3 MCP 接口

`repomix --mcp` 启动一个基于 `rmcp 1.7` 的 stdio JSON-RPC 服务器。四个工具覆盖了库上层和文件系统层的两种主要操作：打包和读取。

### 工具清单

| 工具 | 输入参数 | 输出 |
|---|---|---|
| `pack_codebase` | `directory: String; compress?: bool; include_patterns?: Vec<String>; ignore_patterns?: Vec<String>; top_files_length?: usize; style?: String` | `PackToolResult { description, result, directory_structure, output_id, output_file_path, total_files, total_tokens }` |
| `pack_remote_repository` | `url: String; style?: String` | 同上 |
| `read_repomix_output` | `file_path: String` | 文件内容字符串 |
| `grep_repomix_output` | `file_path: String; pattern: String; context?: usize` | 正则匹配带上下文的行 |

### `PackToolResult` 结构

在这个结果体里，`output_id` 是 MCP 侧为每一次 pack 生成的唯一标识，`output_file_path` 是磁盘上的实际路径，`directory_structure` 是输出中开头的树形结构文本。`result` 是对 `PackResult` 的摘要——包含 `PackMetrics` 的映射，其中 `PackMetrics` 只保留了 `total_files`、`total_tokens`、`total_characters`、`file_token_counts`、`file_char_counts` 和 `top_files_by_tokens`，是精简视角。

### 序列化与错误传播

MCP 协议场景下，工具调用会在 `RepomixMcpServer` 的一个 `lock` 上做互斥访问，防止并发调用互相踩踏。遇到异常时，`anyhow::Error` 会被 `rmcp` 的宏包装成 `rmcp::Error` 回传给 agent 运行时；agent 不必知道失败发生在 `git clone` 阶段还是 `process_files` 阶段，只需消费 `Error` 的 message 即可。

## 5.4 配置文件边界

`RepomixConfig` 是 serde 编解码的双向类型，支持 `repomix.config.json` 从磁盘读取和写入，也支持 `PartialConfig` 作为 CLI 端的增量补丁。

### `PartialConfig` 的作用域

`PartialConfig` 仅携带 CLI 显式提供的键值对，未提供的字段保持 `None`。`RepomixConfig::load` 在合并时，`Some(v)` 会覆盖下层配置，`None` 则视为“用户没表态，保留下层决定”。这确保了 `repomix --compress .` 不会因为不传 `style` 就把样式强行改成默认值，配置层叠的含义是准确的。

### 全局配置位置

全局文件路径为 `~/.repomix/repomix.config.json`（MCP 输出目录为 `~/.repomix/outputs/`），由 `config::global_dir` 模块负责解析与创建，不硬编码到其他 XDG 路径。
