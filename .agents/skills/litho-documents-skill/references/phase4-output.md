| 文件 | 是否必须 | 对应研究来源 |
|------|---------|-----------|
| `{output_dir}/1.概述.md` | 必须 | C1 系统上下文报告 + 预处理报告 |

| `{output_dir}/6.数据库概览.md` | 条件触发 | 数据库概览报告 |

### 数据库文档触发条件（⚠️ 放宽版，对齐 deepwiki-rs）

deepwiki-rs 的 `has_database_files()` 不仅检查 `.sql` 文件，还检查目录名和代码分类。Agent 应使用同样的宽松条件：

**满足以下任一条件即触发数据库文档生成**：
- 存在 `.sql` 或 `.sqlproj` 文件
- 存在 `migrations/`、`sql/`、`db/`、`database/` 目录
- 目录名含 `database` 或 `db`（大小写不敏感）
- 依赖中有 ORM 框架：diesel/sqlx/prisma/typeorm/sequelize/hibernate/sqlalchemy/mongoose 等
- 项目中有 `CodePurpose::Database` 类型的文件（如 migrations、schema 定义）
- 存在数据库配置文件（如 `database.yml`、`dbconfig.json`）

**未触发时**：生成一个极简的 `6.数据库概览.md`，声明"本项目未检测到数据库相关文件"

## 写入顺序

按以下顺序写入（避免依赖未生成的内容）：

1. 创建输出目录：`write_to_file` 写 `1.概述.md`（自动创建目录）
2. 写 `2.架构.md`
3. 写 `3.工作流.md`
4. 创建 `4.Deep-Exploration/` 目录（写第一个模块文件时自动创建）
5. 逐个写入各模块文档 `4.Deep-Exploration/{模块名}.md`
6. 写 `5.边界接口.md`
7. 如适用：写 `6.数据库概览.md`

## Mermaid 图表验证清单

写入每个文档前，对所有 Mermaid 代码块执行以下检查：

### 通用检查
- [ ] 代码块开头使用正确的图类型关键词（`flowchart TD`、`sequenceDiagram`、`C4Context` 等）
- [ ] 所有节点 ID 仅使用字母、数字、下划线（无空格、连字符、特殊字符）
- [ ] 含特殊字符的节点标签用双引号包裹：`A["标签 (含特殊字符)"]`
- [ ] 标签内换行使用 `<br/>` 而非真正的换行符

### flowchart 图检查
- [ ] 箭头语法正确：`-->` 或 `--\|label\|-->`
- [ ] 子图语法：`subgraph 子图名` ... `end`
- [ ] 决策节点用菱形：`{条件?}`（注意加问号）

### sequenceDiagram 检查
- [ ] 所有参与者已声明：`participant A as 名称`
- [ ] 消息语法：`A->>B: 消息` 或 `A-->>B: 返回`
- [ ] `loop` / `alt` / `opt` 块有对应的 `end`

### C4 图检查
- [ ] `C4Context` / `C4Container` / `C4Component` 关键词正确
- [ ] Person/System/Container/Component 定义完整
- [ ] `Rel(src, dst, "label")` 语法正确

### erDiagram 检查
- [ ] 实体名和关系使用正确语法
- [ ] 关系符号：`}|--||`, `}|--|{`, `||--o{` 等

### 常见修复操作

**问题：节点标签含括号或中文冒号**
```
❌ A[用户(User)] --> B[系统: 后台]
✅ A["用户(User)"] --> B["系统：后台"]
```

**问题：节点 ID 含连字符**
```
❌ my-node --> other-node
✅ myNode --> otherNode
```

**问题：子图名含空格**
```
❌ subgraph 输入 层
✅ subgraph inputLayer["输入层"]
```

## 内容质量验证清单

### 叙述风格验证（⚠️ P0 关键新增）
- [ ] 每个章节开头是否有 2-4 句话的叙述性 summary（而非直接跳到表格/列表）？
- [ ] 表格前后是否有解读段落（而非孤立的表格）？
- [ ] 架构决策是否讲了"选择了什么、放弃了什么、为什么"？
- [ ] 是否使用了类比/比喻来解释关键概念？
- [ ] 模块深度文档是否以"这个模块在做什么"开头？
- [ ] 工作流是否先用比喻/叙述建立整体认知？
- [ ] 参数表格是否有"含义解读"而非只有技术描述？
- [ ] 文档读起来像"技术文章"而非"幻灯片提纲"？

### 1.概述.md
- [ ] 包含执行摘要（非空洞描述）
- [ ] 包含用户角色分析（至少 1 个角色）
- [ ] 包含 C4Context Mermaid 图
- [ ] 包含系统范围与边界说明
- [ ] 技术栈信息基于实际代码（非猜测）

### 2.架构.md
- [ ] 包含架构设计理念（具体原则，非通用描述）
- [ ] 包含 C4Container 或等效架构图
- [ ] 包含模块职责表格
- [ ] 包含至少一个时序图或流程图
- [ ] 架构决策基于代码观察（非臆测）

### 3.工作流.md
- [ ] 包含端到端流程图（Mermaid flowchart）
- [ ] 包含时序图（Mermaid sequenceDiagram）
- [ ] 流程步骤对应实际函数/模块（非抽象描述）

### 4.Deep-Exploration/
- [ ] **每个识别到的领域模块都有对应文档**（全覆盖，不得遗漏）
- [ ] 每个模块文档包含内部数据流图
- [ ] 组件表格中引用了**实际文件路径**（每行至少一个）
- [ ] 每个模块文档引用了至少 3 个具体文件路径
- [ ] 每个模块文档引用了至少 2 个具体类型名/函数名
- [ ] 每个模块文档末尾有**置信度评分**（1-10）
- [ ] 文档数量与领域模块报告中识别的模块数一致

### 5.边界接口.md
- [ ] 覆盖了所有对外接口类型（CLI/API/配置等）
- [ ] 每个接口有完整的参数说明
- [ ] 包含使用示例

### 6.数据库概览.md（如生成）
- [ ] 包含 ER 图（Mermaid erDiagram）
- [ ] 表结构描述完整（字段、类型、约束）

## 多语言输出说明

根据 `target_language` 参数调整：

| 语言 | 文件命名 | 界面语言 |
|------|---------|---------|
| `zh`（默认）| `1.概述.md`、`2.架构.md`... | 中文 |
| `en` | `1.Overview.md`、`2.Architecture.md`... | 英文 |
| `ja` | `1.概要.md`、`2.アーキテクチャ.md`... | 日文 |
| `ko` | `1.개요.md`、`2.아키텍처.md`... | 韩文 |

文档正文语言应与目标语言一致，但技术术语（函数名、类名、框架名等）保持原文。
